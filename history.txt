STEP 2 — Agent & Skills framework
--------------------------------------------------------------------------------
Files created:
  agent/__init__.py
  agent/agent.py
  skills/__init__.py
  skills/skill.py
  main.py

Design:
  • Skill base class (skills/skill.py)
      - Abstract base class with abstract method run(**kwargs)
      - to_tool_schema() auto-generates OpenAI function schema from run() signature
        (inspects _fn if set, so the @skill decorator preserves the original sig)
      - Supports Python type annotations → JSON schema types (str/int/float/bool)
      - _fn class attribute: set by @skill decorator to the original function
        so signature introspection works correctly

  • @skill decorator (skills/skill.py)
      - Turns a plain annotated function into a Skill instance
      - Usage: @skill(name="add", description="Add two numbers")
      - Sets _fn = fn so to_tool_schema() sees the real parameter names

  • Agent class (agent/agent.py)
      - OpenAI tool-calling loop
      - __init__(model, system_prompt, client=None, max_iterations=10)
      - client is lazy (created on first use) — safe to instantiate without API key
      - register(*skills) → returns self for chaining
      - run(user_message) → iterates LLM ↔ tool calls until finish_reason="stop"
      - Retry via tenacity (3 attempts, exponential backoff) inside _chat()
      - Supports injecting custom OpenAI-compatible client (Anthropic, local models)

  • main.py — example wiring two skills (@skill decorator) to an agent

Bug fixed during development:
  - @skill decorator originally used staticmethod which broke abstract class
    instantiation → fixed by implementing run() as a proper instance method
    that delegates to fn(**kwargs)
  - to_tool_schema() was inspecting the wrapper run(**kwargs) signature
    instead of the original function → fixed via _fn attribute

STEP 6 — OpenAQ API exploration
--------------------------------------------------------------------------------
Key findings from API exploration:
  • Base URL: https://api.openaq.org/v3
  • Auth: X-API-Key header
  • PM2.5 parameter_id = 2, PM10 parameter_id = 1
  • Data is at sensor level (not location level)
  • Measurements are raw hourly — no server-side daily aggregation
    (period_name=day returns 0 results; must aggregate manually)
  • Active locations identified by datetimeLast != null
  • Endpoint structure:
    GET /v3/countries?parameters_id=2        → countries with PM2.5 data
    GET /v3/locations?parameters_id=2&iso=GB → stations in a country
    GET /v3/sensors/{id}/measurements        → hourly readings (paginated)

STEP 7 — Phase 1 ingestion module
--------------------------------------------------------------------------------
Files created:
  src/openaq_client.py   — API client
  src/pipeline.py        — ingestion pipeline

src/openaq_client.py:
  • OpenAQClient(api_key, requests_per_second=5.0)
  • Built-in throttling (_throttle) to respect rate limits
  • Retry via tenacity (4 attempts, exponential 2–30s)
  • _paginate() handles both numeric and ">1000" found values
  • get_countries(parameters_id)
  • get_locations(country_code, parameters_id, only_active)
  • get_sensor_measurements(sensor_id, date_from, date_to)

src/pipeline.py:
  Outputs:
    data/raw/openaq_locations.csv     — station metadata
    data/processed/openaq_daily.csv   — daily PM2.5/PM10 per location
    data/processed/openaq_monthly.csv — monthly country averages (for WHO join)

  Functions:
    build_locations_df(client, country_codes) → DataFrame
      - Fetches all active locations for PM2.5 and PM10
      - Extracts sensor_id, lat/lon, country, locality

    build_daily_df(client, locations_df, date_from, date_to, max_workers=8)
      - ThreadPoolExecutor (I/O bound) for parallel sensor fetches
      - Aggregates hourly → daily average per location + parameter
      - Filters out negative/invalid readings
      - Logs progress every 50 sensors

    build_monthly_country_df(daily_df)
      - Aggregates daily location data → monthly country level
      - Adds exceeds_who_guideline flag (PM2.5 > 15 µg/m³ WHO 2021 standard)
      - station_count and reading_count for data quality transparency

    run_pipeline(api_key, date_from, date_to, country_codes, max_workers)
      - Orchestrates all three steps
      - Returns dict: {locations, daily, monthly}
      - Defaults: last 365 days, all countries
